# 伙伴四向图集生产规范 v1.0（冻结）

> 适用对象：所有伙伴的四方向行走图。  
> 依赖：[伙伴生产规则](./COMPANION_PRODUCTION_RULES.md) · [美术规范](./ART_BIBLE.md)  
> 目标：不重绘角色、不补画像素的前提下，将生成图稳定转换为清晰、可移动、脚底一致的游戏图集。

## 1. 核心结论

伙伴原图通常是一个大尺寸的 `4 × 4` 网格。每一格除了角色本体，还常含有大面积透明或棋盘背景留白。

禁止直接把整格缩小为 `32 × 32`：

```text
大格（角色 + 大量留白） → 直接缩小 → 角色本体只剩约 20 像素 → 细节丢失
```

正确流程：

```text
原始 4 × 4 图
→ 清除边缘连通的棋盘背景
→ 单格识别角色不透明边界
→ 仅裁切角色本体
→ 全帧按同一比例缩放
→ 水平居中 + 脚底对齐
→ 输出游戏图集
```

缩小不会自动“更清晰”。它只会减少像素；若角色本体在目标格中占用的像素不足，纹理、轮廓和表情都会被采样丢弃。裁掉无效留白的意义，是让有限的目标像素优先留给角色本体。

## 2. 源图交付要求

每种形态必须提供一张未裁切源图：

- PNG；
- 透明背景优先；允许白灰棋盘背景；
- 固定 `4 × 4` 网格；
- 行顺序固定：`front`、`back`、`left`、`right`；
- 每行 4 帧；
- 角色不得越过相邻格；
- 不添加文字、边框、编号、阴影底板或其他 UI；
- 同一形态的四方向使用同一比例与同一脚底高度。

不得只保存已经裁成 `32 × 32` 的结果图。源图是未来修复裁切、升级显示规格和重新验证的唯一依据。

## 3. 产物规格

每张源图输出两个图集：

| 用途 | 单帧尺寸 | 整体尺寸 | 采样 |
| --- | ---: | ---: | --- |
| 紧凑 UI（列表、图标） | `32 × 32` | `128 × 128` | nearest neighbor |
| Pixi 小屋运行时 | `48 × 48` | `192 × 192` | nearest neighbor |

文件名：

```text
{species}_{form}_walk_32_v1.png
{species}_{form}_walk_48_v1.png
```

图集约定：

```text
第 1 行：front_walk_0..3
第 2 行：back_walk_0..3
第 3 行：left_walk_0..3
第 4 行：right_walk_0..3
```

## 4. 裁切与基线规则

1. 仅清除与源图边缘连通的白灰棋盘背景；不得按“白色”一刀切，避免误删奶油色毛发、亮部和水纹。
2. 逐格计算不透明像素边界；不得把整格留白纳入缩放。
3. 一个图集的 16 帧使用**同一个缩放比例**。不能逐帧放大到最大，否则会造成走路时大小跳变。
4. 所有帧必须水平居中，并与同一脚底基线相贴。
5. 保留至少 1 像素安全边距，避免描边贴边被切断。
6. 使用最近邻缩放与 Pixi 最近邻采样；禁止双线性、三次插值和抗锯齿。
7. 不改变角色比例、颜色、轮廓、表情或任何设计像素。

## 5. 小屋显示规则

小屋读取 `48 × 48` 图集，但不把伙伴无节制放大。标准显示尺寸：

| 成长阶段 | 小屋显示尺寸 |
| --- | ---: |
| 初始形态 | `40 × 40` |
| 成长形态 | `48 × 48` |
| 最终形态 | `56 × 56` |

三种尺寸共享脚底基线。成长应当有可见的体量变化，但不能遮挡旅人、家具或破坏小屋空间感。

## 6. 生成后验收清单

每次接入新图集，必须检查：

- [ ] 文件是 `4 × 4`，行顺序和帧序正确；
- [ ] 透明背景无白色棋盘残留；
- [ ] 奶油毛发、灯光、水纹等内部亮部未被误删；
- [ ] 四个方向体量一致；
- [ ] 走路四帧脚底不抖动、不漂浮；
- [ ] 角色不被无意义留白压小；
- [ ] Pixi 使用 nearest neighbor；
- [ ] 32 图集用于紧凑 UI，48 图集用于小屋；
- [ ] 营地肖像与当前成长形态、`evolution_path` 一致；
- [ ] `npm run art:validate` 与 `npm run build` 通过。

## 7. 已知边界

不重绘的前提下，无法从已经被压缩到 `32 × 32` 的旧图中恢复原始细节。因此，旧成品图只能作为兼容资源，不能作为高保真图集的新源。

如需更多细节，必须使用原始未裁切 `4 × 4` 源图重新生成，而不是把旧 32 图放大。
